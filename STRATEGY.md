# OpenBook 内容与产品战略

> OpenBook = "开放式小说" + "启发人们打开书"。
> AI 辅助 + BGM + 读者篇间分支的都市情感内容产品，对标短视频争夺注意力。

---

## 核心哲学

**"会用 AI 但有人味"——去 AI 化。**
平台不奖励"AI 直出"水文，奖励能用 AI 工具但写出自己 voice 的人。
文学性是长期护城河，但**不是入场券**。

---

## 内容战略（2026-06-27 调整）

| | 原定 | 调整后 |
|---|---|---|
| 篇幅 | 600–1500 字微小说 | **2000–4000 字** |
| 内核 | "克制电影感"的文学性 | **情绪浓度优先** |
| 评判标准 | 够不够克制 | **读者读完想不想截图 / 转发** |

**转向原因**：微小说卡在"既没有爽文的情绪刺激，又没有长篇的沉浸感"的中间地带，两头不讨好。短视频用户会停下来读文字，靠的是**情绪浓度极高**（"我要转给某人看"），而不是克制。

**新战略一句话**：**先抓住用户，再谈文学；精品和快销品同在。**

**爆点定位句**：刷 2 分钟会让你心动 / 落泪 / 转发给某人的都市情感故事，配 BGM。

**受众**：短视频消费者——女性 / 年轻白领 / 迁移人群。

**评估一篇故事时首问**：这里有没有一句话，读者会截图转发？

## 创作者模型

- 两类创作者：
  - **素材提供者**：贡献 raw（世界观 / 人设 / 情节），权重 30%
  - **加工者**：用 AI + 提示词写出有 voice 的成品，权重 70%
- 同一人可兼两类；同一条 raw 允许多人加工出不同版本（同源分叉）
- 素材库带打分：加工者投票（采纳次数 + 收藏）为主 + AI 辅助打分
- **"去 AI 化"内化进加工者的核心动作**——交稿前自我守门（见写作宪法）

---

## 写作宪法（加工者交稿前自查五问）

1. 有没有一处**画面**让人想截屏？（电影感）
2. 有没有一处**对白**只有这两个人能说出来？（人物感）
3. 我有没有**替读者下结论 / 替人物表态**？有就删
4. 这篇能不能放出去让陌生人转？（克制 + 不油腻）
5. 这是不是 AI 一眼能看出的味儿？是就改

---

## 读者机制

**篇间分支**（不做篇内）：读完一篇，推荐下一篇时由读者选不同走向。

---

## Phase 0 红线（验证爆点之前不做）

- 不写小程序代码、不自建 CMS
- 不买曲库订阅、不找设计师做 logo / VI
- 不公开素材库投稿（Phase 1 也是邀请制）
- **内容 > 分发 > 工程**，优先级不可颠倒

### 创作工作流（实操路径）

1. 把种子素材导入原型 Creator 的素材库（世界观 / 人设 / 情节）
2. 写作台勾选目标素材 → 调 prompt 生成初稿
3. **加工者手动通读做"去 AI 化"**——按写作宪法五问改
4. 保存进故事库 / 写入 `stories/`
5. 配 BGM（AI 生成 + CC0），导出 H5
6. 在小红书 / 公众号 / H5 投放，看真实读者反应

### 部署架构说明（为什么不是直接部署原型）

原型的 Creator 依赖本地 LLM key、Reader 从 localStorage 读数据——**两者都不适合直接公开部署**（会泄露 key，且新访客看不到内容）。
因此公开面采用**自包含静态 H5**（故事内联、无后端、无 key），托管到 GitHub Pages。原型保持为本地创作工具。

> LLM 提供方：原型走 OpenRouter（OpenAI 兼容）。注意免费 model slug 会轮换 / 下线（`:free` 后缀），失效时在 `.env` 换一个当前可用的免费模型，或绑定自己的直连 key。
